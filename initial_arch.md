### Component Breakdown

#### 1. BCP Xpress Operator + App CRD (The Core)

#### A custom Kubernetes operator that is the single source of truth for the platform. Uses kro (Kubernetes Resource

#### Orchestrator) to compose complex resource graphs from simple inputs.^1

apiVersion: bcpxpress.yourco.io/v
kind: App
metadata:


name: my-service
spec:
itsi: ITSI-
runtime: java-
artifact: nexus.yourco.io/binaries/my-service-1.2.3.jar
port: 8080
replicas: 2
secrets:

- name: DB_URL
  csmPath: csm/my-service/db
  egress:
- host: oracle-db.internal.yourco.io
  port: 1521

#### The operator reconciles this into:

#### Generated Resource Purpose

#### kpack Image Builds OCI image from artifact via CNB

#### Knative Service OR DeploymentRuns the workload (Knative if scale-to-zero desired)

#### Service + Ingress Platform-managed DNS: my-service.apps.yourco.io

#### NetworkPolicy Deny-by-default + declarative egress from spec

#### HPA / Knative autoscaler Autoscaling — no capacity pool problems

#### PodDisruptionBudget Rolling upgrade safety

#### ExternalSecret (from CSM) Secrets injected from CyberArk/Minivault

#### ServiceMonitor Auto-wired Prometheus scraping

#### ConfigMap (OTel collector) OpenTelemetry sidecar config

#### Guardrails enforced by the operator:

#### Resource limits auto-applied (developers can't request 64 GB RAM)

#### Security context locked down (non-root, read-only FS)

#### Only declared egress allowed — everything else denied

#### Image provenance verified

#### 2. Knative Serving — The Runtime Engine

#### Knative Serving is already being explored and is the natural cf push equivalent on Kubernetes.^1

#### Scale to zero — no wasted capacity (solves the CF pool problem)

#### Automatic revisions — push new artifact → new revision → traffic shift

#### Built-in traffic splitting — canary/blue-green without developer effort

#### Cloud-agnostic — runs identically on OpenShift, EKS, AKS, GKE

Developer pushes new JAR
→ kpack builds new image
→ Operator updates Knative Service
→ Knative creates new Revision
→ Traffic gradually shifted (or instant)
→ Old revision kept for rollback

#### This eliminates the need for developers to understand rolling upgrades — it's just how the platform works.^1


#### 3. Platform-Managed Day-One Services

#### These are not optional. Every expert flagged them as adoption blockers if missing.

**Networking**

#### Deny-by-default NetworkPolicy — operator generates from egress spec^1

#### Platform-managed DNS — {app-name}.apps.yourco.io, no IP coupling^1

#### TLS everywhere — cert-manager auto-provisions, developers never touch certs

#### Multi-DC load balancing — platform-level, invisible to developers

**Secrets (CSM Integration)**

#### External Secrets Operator syncing from CyberArk/Minivault^1

#### Developers declare csmPath in app.yaml — secrets appear as env vars

#### No local K8s Secrets visible to developers

**Observability (Built-In, Not Bolt-On)**

#### stdout logging → platform-managed log pipeline (Loki/Splunk)^1

#### OpenTelemetry auto-injected — metrics, traces, logs without developer instrumentation

#### Grafana dashboards auto-generated per app

#### Backstage shows logs/metrics inline — developers never open Grafana directly

**Data Services**

#### External only — platform provides clean binding to external databases, message queues, etc.^1

#### Declared in app.yaml, resolved by operator via CSM + DNS

#### 4. GitOps — ArgoCD

#### ArgoCD (already being explored) handles cluster state reconciliation.^1

Operator writes desired state → GitOps repo
ArgoCD syncs to cluster
Drift detected → auto-corrected

#### This gives you multi-cluster portability. Point ArgoCD at a new EKS cluster → same apps deploy automatically.

#### 5. Backstage — Developer Portal

#### Not just a catalog — it's the single pane of glass replacing cf CLI.^1

#### Onboarding : fill form → scaffolds app.yaml + Git repo + pipeline

#### Deploy : upload artifact or trigger from CI

#### Observe : logs, metrics, deployment history

#### Self-service : scale, restart, rollback — all via UI, zero kubectl


## The Graduation Path

#### The 20% who outgrow the abstraction must have a clean exit.^1

Level 1: App CRD (BCP Xpress) ← 80% of users stay here
│
│ "I need more control"
▼
Level 2: Helm + ArgoCD ← platform-provided charts, still guardrailed
│
│ "I need full K8s"
▼
Level 3: Raw K8s namespace ← full access, platform networking/secrets still available

#### The operator can export a full Kubernetes manifest set from any App CRD — the developer takes that and

#### graduates to Helm or raw YAML without starting from scratch.
